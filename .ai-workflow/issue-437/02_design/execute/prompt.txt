### 本文

## 概要
Jenkins Agentのメモリ使用状況をCloudWatchで監視できるようにする。
アラート設定は不要で、メトリクスの収集と可視化のみを実装する。

## 背景
Jenkins Agentの安定稼働のため、メモリ使用状況を監視する必要がある。
CloudWatch Agentを使用してメモリメトリクスを収集し、CloudWatchコンソールで確認できるようにする。

## 実装内容

### 1. Pulumi: IAM権限の追加
- `pulumi/jenkins-agent-autoscaling/` のIAMロールにCloudWatch Agent用の権限を追加
  - `CloudWatchAgentServerPolicy` マネージドポリシーをアタッチ

### 2. Ansible: CloudWatch Agentのセットアップ
- `ansible/roles/jenkins_agent/` にCloudWatch Agent設定を追加
  - CloudWatch Agentのインストール
  - メモリメトリクス収集設定ファイルの配置
  - CloudWatch Agentサービスの起動・有効化

### 3. 収集するメトリクス
- メモリ使用率（パーセント）
- メモリ使用量（MB）
- メモリ空き容量（MB）

### 4. ⚠️ コスト最適化設定（重要）
スポットインスタンスで頻繁にインスタンスが入れ替わる環境のため、以下の設定で実装する：

**Dimensionは `AutoScalingGroupName` のみを使用**
- インスタンスIDを含めない（デフォルトは含まれる）
- インスタンスが何回入れ替わってもメトリクス数が増えない
- ASG全体の平均値として監視

**コスト比較:**
```
❌ インスタンスIDを含める場合（デフォルト）:
  - 10回入れ替わり → 20メトリクス → 約$6/月

✅ AutoScalingGroupNameのみの場合（推奨）:
  - 何回入れ替わっても → 2-3メトリクス → 約$0.60-1.0/月
```

**CloudWatch Agent設定例:**
```json
{
  "metrics": {
    "namespace": "CWAgent",
    "metrics_collected": {
      "mem": {
        "measurement": [
          {"name": "mem_used_percent"},
          {"name": "mem_used"},
          {"name": "mem_available"}
        ]
      }
    },
    "append_dimensions": {
      "AutoScalingGroupName": "${aws:AutoScalingGroupName}"
    }
  }
}
```

## 設計書作成手順

### 1. 既存コードベース分析

要件定義書とIssue内容から、以下を分析してください：

1. **影響範囲の特定**
   - どのコンポーネント（Ansible/Pulumi/Jenkins/Scripts）が影響を受けるか
   - 既存ファイルの修正が必要か、新規ファイルの作成が必要か
   - 依存関係のあるコンポーネントは何か

2. **既存パターンの調査**
   - 類似の機能実装がないか調査
   - 既存のコーディング規約やパターンを確認
   - 再利用可能なコンポーネントの特定

### 2. 実装戦略の判断（必須）

以下の3つから選択し、**判断根拠を明記**してください：

- **CREATE（新規作成）**: 既存コードとの依存関係がなく、完全に新しい機能
- **EXTEND（拡張）**: 既存コードを拡張・修正する必要がある
- **REFACTOR（リファクタリング）**: 既存コードの構造改善が主目的

**判断基準**：
- 既存ファイルへの影響範囲
- 新規ファイルの作成数
- 既存機能との統合度

**出力形式**：
```markdown
### 実装戦略: CREATE / EXTEND / REFACTOR

**判断根拠**:
- （理由1）
- （理由2）
```

### 3. テスト戦略の判断（必須）

以下から選択し、**判断根拠を明記**してください：

- **UNIT_ONLY**: Unitテストのみ
- **INTEGRATION_ONLY**: Integrationテストのみ
- **BDD_ONLY**: BDDテストのみ
- **UNIT_INTEGRATION**: Unit + Integration
- **UNIT_BDD**: Unit + BDD
- **INTEGRATION_BDD**: Integration + BDD
- **ALL**: すべて

**判断基準**：
- 機能の複雑度
- ユーザーストーリーの有無
- 既存テストとの整合性

**出力形式**：
```markdown
### テスト戦略: UNIT_ONLY / INTEGRATION_ONLY / BDD_ONLY / UNIT_INTEGRATION / UNIT_BDD / INTEGRATION_BDD / ALL

**判断根拠**:
- （理由1）
- （理由2）
```

### 4. テストコード戦略の判断（必須）

以下から選択し、**判断根拠を明記**してください：

- **EXTEND_TEST（既存テストの拡張）**: 既存テストファイルに追加
- **CREATE_TEST（新規テスト作成）**: 新しいテストファイルを作成
- **BOTH_TEST（両方）**: 既存テストの拡張と新規テストの作成

**判断基準**：
- 既存テストファイルの有無
- 既存テストとの関連性

**出力形式**：
```markdown
### テストコード戦略: EXTEND_TEST / CREATE_TEST / BOTH_TEST

**判断根拠**:
- （理由1）
- （理由2）
```
