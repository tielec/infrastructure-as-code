## 概要

`DotFileProcessor`クラスの`_enhance_pulumi_graph`および`_process_node_definition`メソッドが5レベル以上の深いネスト構造を持ち、40以上のメソッドに複雑な状態追跡とURI解析ロジックが分散しています。責務の分離と構造の簡素化により、保守性と可読性の大幅な向上が期待されます。

## 詳細

### 現状の問題点

- `jenkins/jobs/pipeline/infrastructure/pulumi-stack-action/src/dot_processor.py`の397〜473行目において、深いネスト構造（5レベル以上）が存在
- URIパース、ノードラベル生成、依存関係処理などの異なる責務が単一クラスに混在
- 40以上のメソッドに処理が分散しており、全体像の把握が困難
- 複雑な状態追跡ロジックが各所に散在し、バグの温床となっている

### 技術的負債の影響

- コードの理解に多大な時間を要し、新規開発者のオンボーディングを阻害
- 機能追加や不具合修正時のリグレッションリスクが高い
- ユニットテストの作成が困難で、テストカバレッジが低下
- 保守コストの継続的な増大により、技術的負債が蓄積

### リファクタリングの具体的な内容

1. **URIパースロジックの分離**: `UrnProcessor`クラスを新設し、URN/URI解析の責務を集約
2. **ノードラベル生成の独立**: `NodeLabelGenerator`クラスでラベル生成ロジックを一元管理
3. **依存関係処理の分離**: `ResourceDependencyBuilder`クラスで依存グラフ構築を担当
4. **制御フローの簡素化**: ネストした条件分岐を早期リターンパターンで平坦化

### 期待される改善効果

- 各クラスの責務が明確化され、単一責任の原則に準拠
- ネストレベルの削減により、認知的複雑度が大幅に低下
- クラスごとのユニットテストが容易になり、テスタビリティが向上
- コードの再利用性が高まり、将来的な拡張が容易に

## 影響範囲

### 影響を受けるモジュール/ファイル

- **主要対象**: `jenkins/jobs/pipeline/infrastructure/pulumi-stack-action/src/dot_processor.py`
- **新規作成**: `urn_processor.py`, `node_label_generator.py`, `resource_dependency_builder.py`
- **間接的影響**: `dot_processor.py`を利用している上位モジュール（インポート構造の変更）

### 依存関係への影響

- `DotFileProcessor`クラスの公開インターフェースは維持するため、外部からの利用には影響なし
- 内部実装の大幅な変更により、モック・スタブを使用したテストは更新が必要
- 新規クラスへの依存関係が追加されるが、適切な依存性注入により疎結合を維持

### テストへの影響

- 既存の統合テストは動作保証のため全件実行が必須
- 新規クラスごとの単体テストを新規作成（推奨）
- リファクタリング前後での振る舞い同一性を確認する回帰テストの実施

### 破壊的変更の有無

- 公開APIに変更はないため、破壊的変更なし
- 内部メソッドのシグネチャ変更あり（プライベートメソッドのため影響は限定的）

## 推奨される改善策

### 推奨するリファクタリング手法

1. **Extract Class パターン**: 責務ごとに独立したクラスを抽出
   - `UrnProcessor`: URN/URIのパース、正規化、コンポーネント抽出
   - `NodeLabelGenerator`: リソースタイプに応じたラベル生成ロジック
   - `ResourceDependencyBuilder`: 依存関係グラフの構築と検証

2. **Guard Clause パターン**: ネストした条件分岐を早期リターンで簡素化
   ```python
   # Before: 深いネスト
   if condition1:
       if condition2:
           if condition3:
               process()

   # After: 早期リターン
   if not condition1:
       return
   if not condition2:
       return
   if not condition3:
       return
   process()
   ```

3. **Strategy パターン**: ノードタイプごとの処理を戦略クラスとして分離（必要に応じて）

### 参考となるパターンやベストプラクティス

- **Clean Code (Robert C. Martin)**: 関数は一つのことだけを行うべき
- **Refactoring (Martin Fowler)**: Extract Class, Replace Nested Conditional with Guard Clauses
- **SOLID原則**: 単一責任の原則（SRP）の徹底
- **Pythonスタイルガイド**: PEP 8における関数の複雑度推奨値（Cyclomatic Complexity < 10）

## 設計書作成手順

### 1. 既存コードベース分析

要件定義書とIssue内容から、以下を分析してください：

1. **影響範囲の特定**
   - どのコンポーネント（Ansible/Pulumi/Jenkins/Scripts）が影響を受けるか
   - 既存ファイルの修正が必要か、新規ファイルの作成が必要か
   - 依存関係のあるコンポーネントは何か

2. **既存パターンの調査**

### 2. 実装戦略の判断（必須）

以下の3つから選択し、**判断根拠を明記**してください：

- **CREATE（新規作成）**: 既存コードとの依存関係がなく、完全に新しい機能
- **EXTEND（拡張）**: 既存コードを拡張・修正する必要がある
- **REFACTOR（リファクタリング）**: 既存コードの構造改善が主目的

**判断基準**：
- 既存ファイルへの影響範囲
- 新規ファイルの作成数
- 既存機能との統合度

**出力形式**：
```markdown
### 実装戦略: CREATE / EXTEND / REFACTOR

**判断根拠**:
- （理由1）
- （理由2）
```

### 3. テスト戦略の判断（必須）

以下から選択し、**判断根拠を明記**してください：

- **UNIT_ONLY**: Unitテストのみ
- **INTEGRATION_ONLY**: Integrationテストのみ
- **BDD_ONLY**: BDDテストのみ
- **UNIT_INTEGRATION**: Unit + Integration
- **UNIT_BDD**: Unit + BDD
- **INTEGRATION_BDD**: Integration + BDD
- **ALL**: すべて

**判断基準**：
- 機能の複雑度
- ユーザーストーリーの有無
- 既存テストとの整合性

**出力形式**：
```markdown
### テスト戦略: UNIT_ONLY / INTEGRATION_ONLY / BDD_ONLY / UNIT_INTEGRATION / UNIT_BDD / INTEGRATION_BDD / ALL

**判断根拠**:
- （理由1）
- （理由2）
```

### 4. テストコード戦略の判断（必須）

以下から選択し、**判断根拠を明記**してください：

- **EXTEND_TEST（既存テストの拡張）**: 既存テストファイルに追加
- **CREATE_TEST（新規テスト作成）**: 新しいテストファイルを作成
- **BOTH_TEST（両方）**: 既存テストの拡張と新規テストの作成

**判断基準**：
- 既存テストファイルの有無
- 既存テストとの関連性

**出力形式**：
```markdown
### テストコード戦略: EXTEND_TEST / CREATE_TEST / BOTH_TEST

**判断根拠**:
- （理由1）
- （理由2）
```
