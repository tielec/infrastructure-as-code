## 概要

`pr_comment_generator.py`（1985行、97関数）が過度に肥大化しており、統計処理、コメント整形、OpenAI統合、プロンプト生成など複数の関心事が混在しています。単一責任の原則に違反しており、保守性とテスタビリティが著しく低下しているため、早急にモジュール分割を実施する必要があります。

## 詳細

### 現状の問題点
- **ファイルサイズの肥大化**: 1985行という大規模なファイルに97個もの関数が詰め込まれており、コード全体の見通しが極めて悪い状態です
- **関心事の混在**: 統計計算、フォーマット処理、外部API連携、プロンプト構築といった異なる責務が1つのモジュールに混在しています
- **単一クラスへの過度な集約**: 1つのクラスに機能が集中し過ぎており、単一責任の原則（SRP）に違反しています
- **テストの困難性**: 関心事が分離されていないため、ユニットテストの作成と保守が困難です

### 技術的負債の影響
- コードの理解に時間がかかり、新規メンバーのオンボーディングコストが増大します
- バグ修正や機能追加時の影響範囲が予測困難で、デグレードリスクが高まります
- 複数の開発者が同時に作業する際のコンフリクト発生率が上昇します
- リファクタリングを先延ばしするほど、分割コストが指数関数的に増加します

### リファクタリングの具体的な内容
以下の4つのモジュールに分割することを推奨します：

1. **PRCommentStatistics**: 統計データの収集・計算・分析ロジック
2. **CommentFormatter**: コメントの整形・フォーマット処理
3. **PRCommentGenerator**: コア生成ロジック（オーケストレーション層）
4. **OpenAIIntegration**: OpenAI APIとの通信・レスポンス処理

さらに、閾値計算ロジックや関数分類ロジックも専用のユーティリティモジュールとして抽出します。

### 期待される改善効果
- **可読性の向上**: 各モジュールが明確な責任を持つことで、コードの理解が容易になります
- **保守性の向上**: 変更の影響範囲が局所化され、安全な修正が可能になります
- **テスタビリティの向上**: 各モジュールを独立してテストでき、テストカバレッジが改善します
- **再利用性の向上**: 分割されたモジュールを他のコンポーネントから再利用できます

## 影響範囲

### 影響を受けるモジュール/ファイル
- **主要な影響**: `jenkins/jobs/pipeline/docs-generator/pull-request-comment-builder/src/pr_comment_generator.py`
- **依存モジュール**: このファイルをインポートしているすべてのモジュール（主にPRコメント生成パイプライン関連）
- **設定ファイル**: インポートパスの変更に伴う設定ファイルの更新が必要

### 依存関係への影響
- 既存のインポート文（`from pr_comment_generator import XXX`）をすべて更新する必要があります
- 外部から利用されているpublic APIの互換性を維持するため、移行期間中はエイリアスや再エクスポートを検討します

### テストへの影響
- 既存のテストコードのインポートパスとモック対象を更新する必要があります
- モジュール分割により、より細かい粒度でのユニットテストが可能になります
- 統合テストは基本的に変更不要ですが、念のため回帰テストを実施します

### 破壊的変更の有無
- **あり**: インポートパスが変更されるため、依存している全モジュールの修正が必要です
- **緩和策**: 段階的な移行期間を設け、旧インポートパスからの互換性レイヤーを一時的に提供します

## 推奨される改善策

### 推奨するリファクタリング手法
1. **Extract Module パターン**: 関連する関数群を新しいモジュールに抽出します
2. **Facade パターン**: 既存のpublic APIとの互換性を維持するためのファサードを提供します
3. **Dependency Injection**: モジュール間の依存関係を明示的にし、テスタビリティを向上させます

### 段階的な実施計画
**Phase 1: 分析と設計（1週間）**
- 関数間の依存関係を分析し、詳細な分割計画を策定
- 新しいモジュール構造とインターフェースを設計

**Phase 2: 統計モジュール分割（1週間）**
- `PRCommentStatistics`を分離し、統計関連ロジックを移行
- ユニットテストを作成し、機能の同一性を検証

**Phase 3: フォーマッタとAPI統合の分割（1週間）**
- `CommentFormatter`と`OpenAIIntegration`を分離
- 既存のテストを移行し、新規テストを追加

**Phase 4: コアロジック整理と統合（1週間）**
- `PRCommentGenerator`をオーケストレーション層として再構築
- 互換性レイヤーを実装し、既存コードとの共存を実現

**Phase 5: 依存モジュールの更新と検証（1週間）**
- 全依存モジュールのインポートパスを更新
- 統合テストと回帰テストを実施し、本番環境へデプロイ

### 参考となるパターンやベストプラクティス
- **Clean Architecture**: レイヤー間の依存関係を適切に管理し、ビジネスロジックを外部依存から分離します
- **SOLID原則**: 特に単一責任の原則（SRP）と依存性逆転の原則（DIP）を適用します
- **Python PEP 8**: モジュール分割後も一貫したコーディング規約を維持します

## プロジェクト計画書の構成

以下のセクションを含むプロジェクト計画書を作成してください：

### 1. Issue分析
- **複雑度**: 簡単/中程度/複雑を判定
  - 簡単: 単一ファイルの修正、シンプルな機能追加（見積もり: ~4時間）
  - 中程度: 複数ファイルの修正、既存機能の拡張（見積もり: 4~16時間）
  - 複雑: アーキテクチャ変更、新規サブシステム追加（見積もり: 16時間~）
- **見積もり工数**: X時間（根拠を記載）
- **リスク評価**: 低/中/高

### 2. 実装戦略判断

**重要**: このセクションは最も重要です。Phase 2（Design）で使用されるため、必ず明記してください。

#### 実装戦略: CREATE / EXTEND / REFACTOR

**判断根拠**:
- CREATE: 新規ファイル・クラス・モジュールの作成が中心
- EXTEND: 既存コードの拡張（機能追加）が中心
- REFACTOR: 既存コードのリファクタリング（構造改善）が中心

（判断理由を具体的に記載）

#### テスト戦略: UNIT_ONLY / INTEGRATION_ONLY / BDD_ONLY / UNIT_INTEGRATION / UNIT_BDD / INTEGRATION_BDD / ALL

**判断根拠**:
- UNIT_ONLY: ユニットテストのみ（単純なロジック、外部依存なし）
- INTEGRATION_ONLY: インテグレーションテストのみ（外部システム連携中心）
- BDD_ONLY: BDDテストのみ（ユーザーストーリー中心）
- UNIT_INTEGRATION: ユニット + インテグレーション（一般的な機能開発）
- UNIT_BDD: ユニット + BDD（エンドユーザー向け機能）
- INTEGRATION_BDD: インテグレーション + BDD（システム間連携＋ユーザーストーリー）
- ALL: すべてのテストレベル（大規模な変更、高リスク）

（判断理由を具体的に記載）

#### テストコード戦略: EXTEND_TEST / CREATE_TEST / BOTH_TEST

**判断根拠**:
- EXTEND_TEST: 既存テストファイルに追加（既存機能の拡張時）
- CREATE_TEST: 新規テストファイル作成（新規機能追加時）
- BOTH_TEST: 両方（既存テスト拡張 + 新規テスト作成）

（判断理由を具体的に記載）

### 3. 影響範囲分析
- **既存コードへの影響**: 変更が必要なファイル・モジュール
- **依存関係の変更**: 新規依存の追加、既存依存の変更
- **マイグレーション要否**: データベーススキーマ変更、設定ファイル変更等

```markdown

```

**重要**: このチェックリストは、Review Phaseで完了判定に使用されます。各タスクは明確で検証可能な内容にしてください。

**良い例**:
```

```

## 品質ゲート（Phase 0）

作成するプロジェクト計画書は、以下の品質ゲートを満たす必要があります：

- [ ] **実装戦略が明確に決定されている**（CREATE/EXTEND/REFACTOR）
- [ ] **テスト戦略が明確に決定されている**（UNIT_ONLY/.../ALL）
- [ ] **テストコード戦略が明確に決定されている**（EXTEND_TEST/CREATE_TEST/BOTH_TEST）
- [ ] **影響範囲が分析されている**
- [ ] **タスク分割が適切な粒度である**（1タスク = 1~4時間）
- [ ] **リスクが洗い出されている**

これらの品質ゲートは**必須要件**です。作成後、クリティカルシンキングレビューが実施され、品質ゲートを満たさない場合は修正が必要になります。

## 注意事項

1. **実装戦略の判断**: Phase 2で使用されるため、必ず明記してください
   - 実装戦略: CREATE/EXTEND/REFACTOR
   - テスト戦略: UNIT_ONLY/.../ALL
   - テストコード戦略: EXTEND_TEST/CREATE_TEST/BOTH_TEST

2. **具体性**: 曖昧な表現を避け、具体的かつ測定可能な計画を記述
   - NG例: "適切に対応する"、"必要に応じて実施"
   - OK例: "Phase 2で詳細設計を実施"、"最大3回のリトライを実施"

3. **実現可能性**: 見積もりが現実的か、リソースは十分か確認
   - タスク粒度が大きすぎないか（1タスク = 1~4時間）
   - 依存関係が正確に記載されているか

4. **リスク分析の網羅性**: 重要なリスクが見逃されていないか確認
   - 技術的リスク、スコープリスク、リソースリスク、依存リスク

5. **レビューされることを意識**: 作成した計画書はクリティカルシンキングレビューが実施されます
   - 品質ゲート（6つの必須要件）を最優先で満たすこと
   - ブロッカー（次フェーズに進めない問題）がないように注意
   - 改善提案は歓迎されるが、ブロッカーは修正が必須
