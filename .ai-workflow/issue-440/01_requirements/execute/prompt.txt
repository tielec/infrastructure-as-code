### 本文

## 概要

Jenkins Agentのジョブ起動時間を短縮するため、頻繁に使用されるDockerイメージをAMIビルド時に事前にプルしておく。

## 背景

現在、Jenkinsパイプラインでは12種類のDockerイメージを使用している（`jenkins/DOCKER_IMAGES.md` 参照）。
ジョブ実行ごとにDocker Hubからイメージをプルするため、起動に数十秒〜数分かかっている。

### 現在の動作フロー
```
ジョブ実行 → Dockerイメージプル（Docker Hubから） → コンテナ起動 → ジョブ実行
           ↑ この部分に時間がかかる（数十秒〜数分）
```

### 改善後の動作フロー
```
ジョブ実行 → ローカルキャッシュから即座に起動 → ジョブ実行
           ↑ ほぼ即座（数秒）
```

## 期待される効果

### メリット
1. **起動時間の大幅短縮**
   - 小さいイメージ (`python:3.11-slim` 130MB): 10-20秒 → 1-2秒
   - 大きいイメージ (`rust:1.76-slim` 850MB): 1-2分 → 1-2秒

2. **ネットワーク帯域の節約**
   - ジョブ実行ごとのダウンロード不要

3. **安定性向上**
   - Docker Hubのレート制限回避
   - ネットワーク障害の影響を受けにくい

### デメリット
1. **AMIサイズ増加**: 約2-3GB増加見込み
   - EBSストレージコスト: 約\/usr/bin/bash.08/GB/月 × 3GB = 約\/usr/bin/bash.24/月

2. **イメージ更新管理**: イメージを更新するにはAMI再作成が必要

3. **AMIビルド時間増加**: 現在30-45分 → 35-50分程度に増加見込み

## 実装方針

### 1. Component定義に新ステップ追加

`pulumi/jenkins-agent-ami/component-arm.yml` および `component-x86.yml` に以下を追加：

```yaml
- name: PullDockerImages
  action: ExecuteBash
  inputs:
    commands:
      - echo "Pulling Docker images for faster job startup..."
      - systemctl start docker
      - |
        # Python系
        docker pull python:3.11-slim
        
        # Node.js系
        docker pull node:18-slim
        
        # Rust系
        docker pull rust:1.76-slim
        docker pull rust:slim
        
        # AWS系
        docker pull amazon/aws-cli:latest
        
        # Pulumi系
        docker pull pulumi/pulumi:latest
        
        # Ubuntu系
        docker pull ubuntu:22.04
        
        # Python + Node.js複合
        docker pull nikolaik/python-nodejs:python3.11-nodejs20
      - docker images
      - echo "Docker images pre-pulled successfully"
```

### 2. プル対象イメージリスト

`jenkins/DOCKER_IMAGES.md` に記載されている以下の12種類のイメージ：

| イメージ | 使用箇所数 | サイズ（概算） |
|---------|-----------|--------------|
| `python:3.11-slim` | 2 | 130MB |
| `node:18-slim` | 1 | 180MB |
| `rust:1.76-slim` | 2 | 850MB |
| `rust:slim` | - | 850MB |
| `amazon/aws-cli:latest` | 2 | 400MB |
| `pulumi/pulumi:latest` | 1 | 100MB |
| `ubuntu:22.04` | 1 | 77MB |
| `nikolaik/python-nodejs:python3.11-nodejs20` | 3 | 400MB |

**合計**: 約2.9GB

## 要件定義書の構成

以下のセクションを含む要件定義書を作成してください：

### 0. Planning Documentの確認（Planning Phaseが実行されている場合）
- 開発計画の全体像を把握
- スコープ、技術選定、リスク、スケジュールを確認
- Planning Documentで策定された戦略を踏まえて要件定義を実施

### 1. 概要
- Issue本文の「## 概要」セクションを要約
- 背景と目的を明確に記述
- ビジネス価値・技術的価値を説明

### 2. 機能要件
- Issue本文の「## TODO」セクションから機能要件を抽出
- 各要件を明確かつ検証可能な形で記述
- 優先度（高/中/低）を付与

### 3. 非機能要件
- パフォーマンス要件
- セキュリティ要件
- 可用性・信頼性要件
- 保守性・拡張性要件

### 4. 制約事項
- 技術的制約（使用技術、既存システムとの整合性）
- リソース制約（時間、人員、予算）
- ポリシー制約（セキュリティポリシー、コーディング規約）

### 5. 前提条件
- システム環境
- 依存コンポーネント
- 外部システム連携

### 6. 受け入れ基準
- 各機能要件の受け入れ基準（Given-When-Then形式推奨）
- テスト可能な形で記述

### 7. スコープ外
- 明確にスコープ外とする事項
- 将来的な拡張候補

## 品質ゲート（Phase 1）

作成する要件定義書は、以下の品質ゲートを満たす必要があります：

- [ ] **機能要件が明確に記載されている**
- [ ] **受け入れ基準が定義されている**
- [ ] **スコープが明確である**
- [ ] **論理的な矛盾がない**

これらの品質ゲートは**必須要件**です。作成後、クリティカルシンキングレビューが実施され、品質ゲートを満たさない場合は修正が必要になります。

## 注意事項

1. **具体性**: 曖昧な表現を避け、具体的かつ測定可能な要件を記述
   - NG例: "適切に処理する"、"必要に応じて対応"
   - OK例: "3秒以内にレスポンスを返す"、"エラー発生時は管理者に通知"

2. **整合性**: 各セクション間で矛盾がないか確認
   - 機能要件と受け入れ基準が対応しているか
   - 非機能要件と制約事項が矛盾していないか

3. **完全性**: Issue本文の情報を漏れなく反映
   - TODOセクションの項目をすべて機能要件に反映
   - 概要セクションの背景・目的を明確に記述

4. **検証可能性**: 各要件が検証可能（テスト可能）な形で記述
   - 受け入れ基準は Given-When-Then 形式で記述
   - 成功条件・失敗条件が明確に識別可能

5. **レビューされることを意識**: 作成した要件定義書はクリティカルシンキングレビューが実施されます
   - 品質ゲート（4つの必須要件）を最優先で満たすこと
   - ブロッカー（次フェーズに進めない問題）がないように注意
   - 改善提案は歓迎されるが、ブロッカーは修正が必須
